## 1. 추천 → 입어보기 시 추천한 옷이 자동 선택되지 않는 버그

### 문제

추천 탭에서 조합 카드의 "입어보기" 버튼을 누르면 `/(tabs)/fitting` 으로 이동하면서 `params.ids` 가 함께 전달된다. 그런데 fitting 탭에 들어가도 추천한 옷들이 선택된 상태로 보이지 않았다. 사용자가 "추천 그대로 입어볼게" 의도로 누른 동작이 실제론 빈 선택 화면을 띄우고, 다시 옷을 골라야 했다.

처음엔 `params.ids` 자체가 안 넘어오는 줄 알았지만, 로그를 보니 정상 전달되고 있었다.

### 해결

`fitting.tsx` 의 selectedIds 초기화 코드를 다시 봤다.

```tsx
const [selectedIds, setSelectedIds] = useState<string[]>(() => parseIds(params.ids));
```

`useState(() => ...)` 의 lazy initializer 는 **컴포넌트 최초 마운트 1회만** 실행된다는 것을 짚어냈다. Expo Router의 탭 네비게이션은 탭을 전환해도 화면을 unmount 하지 않는다. 즉 추천 탭에서 처음 fitting 탭으로 갔을 때 1회 적용된 후, 다시 추천 → 입어보기 → 새 ids 가 들어와도 lazy initializer 가 재실행되지 않아 무시되고 있었다. 

`useEffect([params.ids])` 로 갱신 로직을 추가했고, 동시에 `router.setParams({ ids: undefined })` 로 ids 를 비워 두는 단계를 넣었다. 이는 사용자가 fitting 탭에서 직접 선택을 해제한 뒤 다른 탭에 다녀와도, 옛 ids 가 다시 강제 적용되어 선택이 살아나는 부수 버그를 막는 가드다.

```diff
+ useEffect(() => {
+   const incoming = parseIds(params.ids);
+   if (incoming.length === 0) return;
+   setSelectedIds(incoming);
+   router.setParams({ ids: undefined });
+ }, [params.ids, router]);
```

### 결과

- 추천 → 입어보기 → fitting 탭 진입 시 옷이 미리 선택된 상태로 표시된다.
- 사용자가 직접 해제 후 탭 전환을 반복해도 옛 추천 선택이 되살아나지 않는다.
- 타입체크 및 Metro HMR 자동 리로드로 즉시 검증.

근본 원인이 React 의 `useState` lazy initializer 동작 + Expo Router 의 탭 화면 보존 동작의 교차점에 있었다. 표면 증상("ids 가 전달이 안 되네")만 보고 라우팅 로직을 의심했다면 한참 헤맸을 것이다.

## 2. 옷장 사진 1장에서 여러 벌 동시 인식 기능 통합

### 문제

기존 등록 흐름은 옷 1벌 = 사진 1장 기준으로 설계돼 있었다. 옷장 전체 사진 1장에 옷 9벌이 보일 때 사용자는 사진 9장을 따로 찍거나 잘라서 등록해야 했다. 한편 `scripts/experiments/multi_detect.py` 라는 별도 트랙 실험 코드는 이미 존재했지만, 본 앱과 통합되지 않은 상태였다.

### 해결

실험 코드를 본 코드로 끌어오는 것에서 멈추지 않고, **검토 단계까지 포함한 새로운 등록 플로우 한 갈래**를 신설했다. AI 가 감지한 결과를 사용자가 그대로 받지 않고, 체크박스로 골라 등록하게 만들었다(잘못 감지된 객체 — 예: 백팩 — 를 사용자가 한 번에 거를 수 있도록).

서버 측:
- `POST /detect-multi` 엔드포인트 신설 (`server/routers/detect.py`)
- Gemini Flash 멀티 감지 전용 프롬프트 (`server/prompts/detect_multi.py`) — `box_2d` 정규화 좌표(0~1000) 출력 강제
- 라우터에서 카테고리/박스 좌표 검증, 잘못된 항목은 조용히 skip
- pytest 6개 (정상 응답, 잘못된 박스 필터, 잘못된 카테고리 필터, 빈 파일 거부, 잘못된 JSON 거부, 파일 누락 422)

앱 측:
- `services/api.ts` 에 `detectMultipleClothes()` 추가
- `services/imageUtils.ts` 에 `cropAndSaveClothingImage()` 추가 — `box_2d` 정규화 좌표를 `Image.getSize` 로 실제 픽셀로 변환 후 `expo-image-manipulator` 로 crop+resize+save
- 신규 화면 `app/clothing-detect-review.tsx` — 감지된 옷들을 썸네일 그리드로 띄우고 체크박스로 선택, 선택된 항목만 등록 폼으로 넘김
- 옷장 시트에 "한 장에서 여러 벌 인식" 버튼 추가 → 갤러리에서 1장 선택 → 검토 화면으로 이동

검증 과정에서 서버 reload 가 새 라우터를 안 잡는 이슈가 있어 옛 uvicorn 프로세스 2개를 정리하고 재시작했고, 이후 침대 사진에 직접 호출해서 9벌 모두 정상 분류되는 것을 confidence 0.92~0.99 로 확인했다.

### 결과

- 서버 pytest: 18 / 18 PASS
- 앱 jest: 28 / 28 PASS, tsc --noEmit clean
- 9벌 사진에서 [상의 4 + 하의 4 + 악세사리(백팩) 1] 정상 분류 (실험 grid 캡처로 검증)
- 등록 시간 단축: 9장 개별 등록 → 1장 + 체크박스 선택

사용자 검증 단계에서 Discord 첨부 권한이 없어 사진을 못 받는 부수 마찰이 있었지만, 두 가지 우회(권한 추가 / 로컬 경로 공유)를 제시해서 막히지 않았다.

## 3. 추천 카드의 한국어 코멘트가 실제 가상 피팅 결과에 반영되지 않는 문제

### 문제

추천 카드는 "포멀한 출근룩", "캐주얼한 데이트룩" 같은 한국어 한 줄 코멘트를 보여준다. 그런데 사용자가 카드의 "입어보기" 를 누르면 가상 피팅(Gemini Pro 이미지 편집)은 옷 이미지만 받아 합성할 뿐, 그 코멘트의 *어떻게 입어야 그 인상이 나오는지* — 셔츠 턱인, 소매 롤업, 오버핏, 단추 상태 같은 디테일 — 을 모른다. 결과 이미지는 옷이 어색하게 얹혀 있거나 핏이 캐릭터의 인상과 맞지 않는다.

근본은 `/recommend` 가 "사용자에게 보여줄 텍스트"만 만들고 끝나는 데 있었다. 모델에 줄 *지시문*은 따로 만들지 않았다.

### 해결

먼저 결정해야 할 게 4가지였다. 각각을 단답으로 정리해서 차례로 의사결정.

| 결정 | 선택 | 이유 |
|------|------|------|
| Q2: 사용자에게도 보여주나 | C 분리 | 한 텍스트가 카드용+프롬프트용 두 역할 다 하면 둘 다 어중간해짐. 응답 필드 하나 늘리는 비용은 크지 않음 |
| Q3: 추천 → 입어보기 동선 | A 결과 직행 | fitting 탭 경유 시 옷이 바뀌면 styling_prompt 가 어긋남. 사용자 의도("이 코디 그대로")와 일치 |
| Q4: 수동 경로도 styling 받나 | A 추천 경로만 | 추가 LLM 호출 없이 범위 최소. 추후 확장 가능 |
| Approach | A 단일 호출 + URL 파라미터 | 인메모리 store 도입(B) 이나 별도 lazy 호출(C) 보다 변경 면적·지연이 가장 작음. URL 길이는 200~400자 캡으로 충분 |

여기서 그치지 않고 **설계안을 4섹션으로 쪼개 각각 사용자 승인을 받은 뒤 진행**했다. 한 번에 큰 PR을 던지면 잘못된 가정이 후반에 발견될 위험이 컸기 때문.

- 섹션 1: 서버 `/recommend` — 스키마(`RecommendCombination.styling_prompt`) + 프롬프트 확장 (영어 1~3문장, 200자 내외, 포즈/카메라/배경 언급 금지)
- 섹션 2: 클라 라우팅 — `tryOn` 4번째 인자, `recommend.tsx` 의 `handleTryOn` 이 `/fitting-result-new` 로 직행, URL 파라미터 키 `sp`
- 섹션 3: 서버 `/try-on` — `Form("")` 으로 받아 400자 trim 후 `TRYON_PROMPT` 끝에 `"Styling guide (apply naturally without changing pose, face, or background):"` 헤더와 함께 결합. 빈 문자열이면 결합 자체를 건너뛰어 수동 경로 100% 백워드 호환.
- 섹션 4: 테스트 + 검증 + 커밋 분할 (4커밋, 서버 → 앱 순)

구현은 TDD 로 — 먼저 서버 테스트 7개를 추가해 모두 빨강을 확인한 뒤 라우터/스키마/프롬프트를 고쳐 초록으로 만들고, 앱 쪽도 jest 테스트 후 구현 순으로 진행.

### 결과

- server pytest: 24 / 24 PASS (기존 18개 + 새 6개 + 기존에 미보장이던 styling-guide 미포함 assert 추가)
- app jest: 30 / 30 PASS
- app `tsc --noEmit`: 0 에러
- 4커밋 분할 (`feat-mvp` 브랜치):
  - `4c0f066` feat(server): /recommend 응답에 styling_prompt 추가
  - `ea494cc` feat(server): /try-on 에 styling_prompt 수신 + 프롬프트 결합
  - `49fa036` feat(app): tryOn 시그니처 + RecommendCombination 타입 확장
  - `0ed6ed9` feat(app): 추천→입어보기 결과 직행 + styling guide 전달

부수효과로 사용자 클릭 1회가 줄었다(추천 → fitting 탭 → 피팅하기 → 결과 → 추천 → 결과). 수동 피팅 경로는 styling guide 가 빈 문자열로 흘러 기존 동작과 동일.

## 4. 미커밋 변경이 섞여 있는 파일을 기능 단위로 깔끔하게 분할 커밋

### 문제

styling_prompt 작업을 시작할 때 working tree 에는 이미 별 트랙(멀티디텍트 통합 + SafeArea 수정 + .gstack/ gitignore + 기타) 의 미커밋 변경이 한 무더기 쌓여 있었다. 그 위에 styling_prompt 변경을 더하니, 일부 파일은 두 작업의 변경이 한 파일 안에 섞이는 상태가 됐다.

| 파일 | 멀티디텍트 변경 | styling_prompt 변경 |
|------|----------------|--------------------|
| `server/schemas.py` | `DetectMultiItem` 클래스 추가 | `RecommendCombination.styling_prompt` 추가 |
| `app/services/api.ts` | `detectMultipleClothes`, `DetectMultiItem`, 타임아웃 상수 | `tryOn` 4번째 인자, `RecommendCombination` 인터페이스 확장 |
| `app/__tests__/api.test.ts` | 타임아웃 테스트 16_000 → 61_000 수정 | styling_prompt 관련 신규 테스트 |

`git add -p` 의 -p 플래그도 본질적으로 인터랙티브여서 비대화형 환경에서 제어가 까다로웠다. 그렇다고 두 트랙을 하나의 커밋으로 묶으면 커밋 메시지가 거짓말이 된다.

### 해결

기능 단위 백업 → HEAD 복원 → 재적용 → 커밋 → 백업 복원 패턴을 적용했다.

```bash
# 1) 혼합 파일 백업
cp server/schemas.py /tmp/pcbak/schemas.py
cp app/services/api.ts /tmp/pcbak/api.ts
cp app/__tests__/api.test.ts /tmp/pcbak/api.test.ts

# 2) HEAD 버전으로 복원 (멀티디텍트 + styling 둘 다 사라짐)
git checkout HEAD -- server/schemas.py app/services/api.ts app/__tests__/api.test.ts

# 3) Edit 으로 styling_prompt 변경만 다시 적용

# 4) 그 파일 + 순수 styling 파일들을 stage + commit

# 5) 백업으로 복원 (멀티디텍트 변경이 다시 working tree 에 살아남)
cp /tmp/pcbak/schemas.py server/schemas.py
```

핵심은 5단계의 복원이다. 4단계 commit 으로 새 HEAD 가 styling_prompt 까지 포함하게 됐으므로, 백업(둘 다 포함)을 working tree 로 덮어쓰면 git status 는 *멀티디텍트 변경만* 미커밋으로 남는다. 다음 커밋 배치도 이 방법을 반복.

### 결과

- styling_prompt 4커밋이 깔끔하게 land. 각 커밋의 diff 가 메시지와 정확히 일치.
- 별 트랙(멀티디텍트 통합 등) 미커밋 변경은 그대로 보존 — 사용자 승인 전이라 의도적으로 남김.
- 커밋 후 검증 (`pytest`, `jest`, `tsc`) 모두 통과해 분할 과정에서 누락 없음을 확인.

대안으로 `git stash --keep-index` 나 인터랙티브 patch 도 고려했지만, 백업-복원 패턴이 가장 직관적이고 실수 시 되돌리기도 쉬웠다. 비슷한 상황을 또 만나면 재사용할 수 있는 표준 절차로 굳혔다.

## 종합

오늘 4개 작업을 통해 다음을 적용/개선했다.

- **버그**: useState lazy initializer + 탭 화면 보존이 만든 동작 미스매치를 표면 증상이 아니라 React/Expo Router 동작 원리에서 짚어 한 줄 useEffect 로 고정.
- **기능**: 옷 등록 UX 의 병목(N장 등록 → 1장 + 체크박스)을 서버 새 엔드포인트, 새 검토 화면, box_2d crop 헬퍼까지 종단 통합.
- **품질**: 추천이 만든 *의도* 가 가상 피팅 *결과* 까지 흐르도록 데이터 모델·라우팅·프롬프트를 한 번에 정렬. 수동 경로는 빈 문자열 가드로 회귀 0.
- **운영**: 두 트랙이 섞인 작업 트리를 깨지지 않게 4커밋으로 분할하는 절차를 표준화.

검증 합계 — server pytest 24 PASS, app jest 30 PASS, app tsc clean.
