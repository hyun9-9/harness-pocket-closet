## 2. Supabase 의 새 비대칭(ECC P-256) signing key 시스템을 phase 1 plan 적용 단계에서 짚어내기

### 문제

phase 1 plan 은 `SUPABASE_JWT_SECRET` (HS256 shared-secret) 을 env 로 받아 서버에서 토큰 검증한다고 가정. 그런데 사용자가 키 등록 후 보여준 화면이 **JWT Keys 페이지** 였고, 거기 "Current key" 가 ECC P-256 (비대칭) + "Previous key" 가 Legacy HS256 (3시간 전 rotated). 즉 이 프로젝트는 이미 새 비대칭 시스템으로 마이그레이션돼 있어서 옛 흐름이 통하지 않는다.

만약 plan 을 그대로 따라서 HS256 shared-secret 을 받아 검증을 짠다면, 새로 발급되는 모든 토큰은 ECC 로 서명되므로 검증이 100% 실패. 그러나 표면적으로는 "키만 받아 채우면 된다" 라는 평탄한 단계로 보였다.

### 해결

내가 **Supabase 의 키 시스템 두 세대를 짚어내고 plan 을 비대칭 검증 흐름으로 비틀었다**. 정확히는:

1. 먼저 사용자가 "어떤 거 받아?" 라고 묻길래 키 화면 위치 (Project Settings > API Keys vs JWT Keys) 를 분리해 안내. 그 과정에서 사용자가 보여준 JWT Keys 페이지가 ECC 로 마이그레이션된 상태임을 인식.
2. Plan 의 `SUPABASE_JWT_SECRET` 환경변수를 빼고, **JWKS endpoint 흐름** 으로 갈음. `SUPABASE_URL` 만 받아서 `${SUPABASE_URL}/auth/v1/.well-known/jwks.json` 으로 자동 조립 → `PyJWKClient` 가 키 캐싱 + ES256/RS256/HS256 다 허용해서 Supabase 가 어떤 마이그레이션 단계든 다 통과하게.
3. config.py 의 Settings dataclass 에 `supabase_jwks_url` 프로퍼티만 추가. 환경변수 1개 줄임.

```python
@property
def supabase_jwks_url(self) -> str:
    return f"{self.supabase_url.rstrip('/')}/auth/v1/.well-known/jwks.json"
```

이게 plan 대비 의도적 deviation 이라 commit 메시지에 명시 — 이유 (이 프로젝트는 새 비대칭) + 결과 (env 변수 1개 줄어듦, 미래 호환).

### 결과

- phase 1 land — `4a2ffe4`. 12 파일 +696/-4. supabase migrations + storage 가이드 + docs/supabase-setup.md + 의존성 + scheme.
- 회귀 0: server pytest 24 / app jest 30 / tsc clean
- phase 2 의 JWT 미들웨어가 그대로 JWKS 비대칭으로 land 됨 (`f1932a8`) — pyjwt[crypto] 의 PyJWKClient + ES256 검증. 테스트도 ECC P-256 키페어를 fixture 로 만들어 ES256 으로 sign 후 fake JWKS client 가 public key 반환하는 패턴 — production 흐름과 같은 알고리즘 경로
- pytest 40 (기존 24 + auth 8 / uploads 5 / users-bootstrap 3)

만약 plan 을 그대로 따라 HS256 으로 짰다면 phase 3 의 OAuth 직후 첫 Bearer 호출에서 검증 100% 실패. plan 은 항상 한 시점의 가정이라 — *현장 상태와 충돌하면 plan 을 바꾸는 게 빠르다* 는 걸 다시 확인.

## 3. Google OAuth 흐름 4단 디버깅 — Supabase Site URL fallback 부터 SecureStore 2KB 한도까지

### 문제

phase 3 land 후 사용자가 "Google 로그인 누르면 페이지 뜨고 동작 없어" 라고 보고. 표면 증상은 단순했지만 실제로는 **다른 단계의 4개 문제가 직렬로 누적**돼 있었다. 하나씩 풀때마다 새 증상이 드러나는 양파 구조.

각 단계의 증상:

1. **계정 선택 후 빈 페이지 (`localhost:3000/#access_token=eyJ...`)** — Google + Supabase 토큰 발급은 됐는데 redirect 가 default Site URL (localhost:3000) 로 떨어짐. 폰엔 그 주소 reach 안 됨.
2. **redirect URL 화이트리스트 미등록** — Supabase Dashboard 의 Redirect URLs 가 비어있어 `redirectTo: exp://...` 가 무시되고 fallback 발동.
3. **계정 선택 후 무한 로딩** — 화이트리스트 등록 후에도 멈춤. WebBrowser.openAuthSessionAsync 가 redirect 를 받아 자동 close 해야 하는데 안 됨.
4. **앱이 "Something went wrong" crash** — OAuth 통과 후 setSession 단계에서 SecureStore 가 2048 byte 초과 값 저장 fail → 앱 throw.

### 해결

각 단계마다 가설 → 검증 → fix 를 1인칭으로 진행했다.

**단계 1 — Site URL fallback 짚기**: 첫 스크린샷의 URL bar `localhost:3000/#access_token=eyJ...` 에서 토큰이 박혀있는 걸 보고 "토큰 발급은 됐는데 redirect URL 매칭이 실패해서 default Site URL 로 fallback 했다" 는 가설로 좁혔다. Supabase 의 Site URL 이 fallback 이라는 docs 동작을 알고 있어야 짚을 수 있는 부분.

**단계 2 — Redirect URLs 분리 등록**: 사용자가 처음엔 3개 URL 을 한 줄에 다 붙여넣어서 `Total URLs: 1` 의 잘못된 entry 가 됨. 화면을 보고 "각각 따로 Add URL" 안내. wildcard 는 `*` 가 아니라 `**` (double star).

**단계 3 — 외부 브라우저 (Samsung Internet) 의 Custom Tabs 의심 → curl 로 서버 단 검증 → maybeCompleteAuthSession 누락 짚기**: `curl -sS -D -` 로 `/authorize?provider=google&redirect_to=...` 직접 호출. HTTP 302 + Location header 가 정상 Google OAuth URL 임을 확인 (Supabase 측은 정상). 그러면 클라 측 — Custom Tabs 가 redirect 후 닫히지 않는 안드로이드 케이스 의심. 사용자가 보내준 블로그 글 (https://sensol2.tistory.com/113) 과 우리 코드 비교 — 차이는 한 줄, **`WebBrowser.maybeCompleteAuthSession()` 의 모듈 top-level 호출 누락**. AuthContext 상단에 한 줄 추가 + makeRedirectUri (expo-auth-session 표준) + flowType 'pkce' 로 전환 + Linking.addEventListener('url') 와 WebBrowser 결과를 Promise.race 해서 둘 중 먼저 도착하는 쪽으로 (Custom Tabs 가 안 닫혀도 OS 가 Expo Go 로 launch 하면 url event 가 먼저 옴).

```ts
// AuthContext.tsx top-level
WebBrowser.maybeCompleteAuthSession();

// signInWithGoogle 안
const wbPromise = WebBrowser.openAuthSessionAsync(data.url, redirectUrl).then(...);
const deepLinkPromise = new Promise<string>((resolve) => {
  resolveDeepLink = resolve;
});
const linkingSub = Linking.addEventListener('url', (event) => {
  if (event?.url) resolveDeepLink(event.url);
});
const resultUrl = await Promise.race([wbPromise, deepLinkPromise]);
```

**단계 4 — SecureStore 2KB 한도 + chunking adapter**: Metro 로그에서 `WARN Value being stored in SecureStore is larger than 2048 bytes` 발견. Supabase v2 session 토큰이 그보다 큼. SecureStore 의 set 이 fail 하면 supabase storage adapter 가 throw → 앱 crash. Supabase 공식 RN 가이드의 chunking adapter 패턴 적용 — `{key}_size` 에 chunk 수, `{key}_0..{n-1}` 에 2000 byte 단위로 쪼개 저장. 기존 chunk 정리 로직 포함.

### 결과

- phase 3 land + 4단 fix 가 한 commit 으로 (`4eff6c4`) — 119 insertions / 32 deletions
- 사용자 실기기에서 Google 로그인 → 옷장 진입 정상 (스크린샷으로 9벌 다 표시 확인)
- jest 34 / tsc 0 / pytest 40 회귀 0
- ADR-013 에 흐름 명시 — Supabase Hosted OAuth 위임 + JWKS 검증 + Custom Tabs 회피 패턴

표면 증상 ("페이지 뜨고 동작 없어") 만 보고는 절대 풀 수 없는 케이스. *서버 측 응답을 curl 로 검증해 클라/서버 책임을 분리* 한 단계가 가장 효과적이었다. Supabase 가 정상 302 를 주는 걸 확인하고 나니 클라 쪽으로 좁혀졌다.

## 4. Phase 4 의 `/clothes` 404 — uvicorn ghost worker process 잡기

### 문제

phase 4 의 첫 옷 sync 시도에서 사용자 폰 토스트가 `[debug] sync failed: clothes-upsert 실패 (status 404): {"detail":"Not Found"}`. 서버 코드와 main.py 는 정상이고 테스트도 통과 (pytest 48). 그런데 `curl http://localhost:8000/openapi.json | jq '.paths | keys'` 결과에 `/clothes` 자체가 없음.

`uvicorn --reload` 가 `routers/clothes.py` 변경을 감지해서 reload 했다는 로그는 있는데 (`WatchFiles detected changes...`), 실제 in-memory 의 라우터 등록은 옛 상태. 즉 reloader child process 가 분리됐고 새 worker 가 import 만 다시 했지 routes 가 안 들어옴 — Windows 의 ghost worker.

### 해결

`TaskStop` 으로 background uvicorn 종료했지만 health endpoint 는 여전히 응답. 즉 process 가 죽지 않음.

내가 process 추적을 단계별로 진행했다:

1. `netstat -ano | grep ":8000"` — PID 9316, 10120 이 LISTENING
2. `taskkill //F //PID 9316 //PID 10120` (git bash 의 // escape 사용) — "종료되었음" 응답
3. 그래도 netstat 이 같은 PID 표시 — Windows netstat 캐시
4. `tasklist //FI "IMAGENAME eq python.exe"` — PID 1748, 34460 의 python.exe 살아있음 (얘들이 실제 worker)
5. 그 PID 두 개 강제 종료 → port 해방 확인
6. 새 uvicorn 시작 시 `--reload` 빼고 그냥 `uvicorn main:app --host 0.0.0.0 --port 8000` — reloader / worker 분리 회피

```bash
taskkill //F //PID 1748
taskkill //F //PID 34460
# 그 후
uvicorn main:app --host 0.0.0.0 --port 8000  # --reload 없이
```

`--reload` 가 worker 재기동을 보장하지 못하는 케이스에서는 그냥 깔끔하게 새로 띄우는 게 Windows 환경에서 빠르다. 또 background task 의 cleanup 이 자식 프로세스까지는 안 잡는다는 걸 확인 — TaskStop 응답을 그대로 신뢰하지 말고 `tasklist` 로 한번 더 검증해야 한다.

### 결과

- 새 uvicorn 띄우자 `/openapi.json` 의 paths 에 `/clothes`, `/clothes/upsert`, `/clothes/{clothing_id}` 정상 노출
- 사용자 재시도 → DB 직접 쿼리에서 옷 9벌 (상의 4 / 하의 4 / 악세사리 1) push 확인. user_id = `1be1d024-...`, image_url = `{user_id}/{uuid}.jpg` 패턴 정상.
- 디버그 토스트 제거 후 `f6f177e` 로 정리

`uvicorn --reload` 의 신뢰성을 Windows 에서 의심하는 게 좋다는 학습. 같은 상황 또 만나면 1) tasklist 로 worker PID 확인 2) taskkill /F 3) `--reload` 없이 재시작 — 3단 절차로 표준화.

## 5. 이미지 sync 의 LWW 와 multipart fetch 의 충돌 — 두 단 fix

### 문제

phase 4 옷 sync 가 검증 통과한 후 phase 5 까지 land. 그런데 사용자가 "피팅하기 누르자 마자 Network request failed". `/try-on` 호출이 서버까지 도달조차 안 함 (서버 로그에 흔적 없음).

처음엔 LAN 끊김 의심. 사용자가 "또 똑같이 누르자 마자 떠" 라고 함 — 즉 fetch 가 시작 단계에서 throw. multipart 인 try-on 에서 fetch 는 이미지 file URI 를 읽어 multipart 로 보내는데, **URI 가 file:/// 또는 http:// 가 아닌 형태면 RN 이 invalid URL 로 즉시 throw**.

그러면 어디서 invalid URL 이 들어왔나? phase 4 의 LWW 머지가 push 직후 pull 시 같은 row 를 받아 (서버가 trigger 로 updated_at 갱신해서 since 보다 살짝 미래) 로컬을 덮어씀. 그 과정에서 `imageUri` 가 file:/// 에서 remote path (`user-A/abc.jpg`) 로 바뀜. 이후 try-on 의 fetch 가 그 path 를 그대로 받아 fail.

### 해결

두 단으로 fix 했다.

**1단 — storage LWW 에서 file:// imageUri 보존 (`7109350`)**: `upsertClothesFromRemote` / `upsertFittingsFromRemote` 의 LWW 머지에서 `local.imageUri` 가 `file:` / `content:` prefix 면 그것을 보존, 다른 필드만 remote 로 머지. 표시는 ClothingCard 가 어차피 remote path 도 signed URL 로 변환하니까 외부 동작 동일.

```ts
if (remoteTs > localTs) {
  byId.set(remote.id, {
    ...local,
    ...remote,
    imageUri: isLocalUri(local.imageUri) ? local.imageUri : remote.imageUri,
    remote_synced_at: remoteTs || nowIso(),
  });
}
```

그러나 사용자가 다시 시도해도 여전히 같은 에러. 이미 storage 에 remote path 로 박힌 옷들은 LWW fix 가 못 잡음 — 한번 망가진 storage 는 그대로다.

**2단 — tryOn 호출 시점에 robust 변환 (`0bddfb0`)**: api.ts 의 `tryOn` 안에서 이미지 URI 를 호출 직전에 검사 — `file:/content:/http(s):/data:` 면 그대로, 그 외 (= remote path) 면 `getReadUrl(bucket, uri)` 로 5분 캐시된 signed read URL 받아 https URL 로 변환. multipart fetch 가 https URL 받으면 RN 이 정상으로 byte 가져옴.

```ts
async function resolveFetchableUri(uri: string, bucket: Bucket): Promise<string> {
  if (FETCHABLE_RE.test(uri)) return uri;
  return getReadUrl(bucket, uri);
}

// tryOn:
const [resolvedPerson, resolvedClothes] = await Promise.all([
  resolveFetchableUri(personUri, 'person'),
  Promise.all(clothingUris.map((u) => resolveFetchableUri(u, 'clothes'))),
]);
```

추가로 **3단 — applyRemoteProfile 에도 같은 보존 적용 (`b16213e`)**. 사용자가 "전신사진 찍었는데 reload 후 사라져" 라고 보고. 같은 LWW 버그가 profile 에도 있었던 것. person camera 화면이 remote path 를 표시 못 하니 "사라진 것처럼" 보였다. applyRemoteProfile 에서 personImageUri 가 file:// 면 보존하도록 동일 패턴 적용.

### 결과

- 피팅하기 → 옷 5벌 합성 정상 (Gemini Pro, 60s 내)
- 전신사진 등록 후 reload 해도 유지
- jest 58 (storage / clothesSync / fittingsSync / profileSync 다 통과)
- ADR-014 에 "multipart fetch 시 imageUri 가 remote path 면 호출 시점에 signed read URL 로 변환 — 로컬 file:// 캐시가 있으면 그대로" 라고 명시

표면 증상 ("Network request failed") 만 보고 LAN 의심으로 갔다면 한참 헤맸을 거다. *서버 로그에 호출 자체가 없다* → 클라 측 fetch 가 시작 안 됨 → URI 형태 의심 — 단계 좁히는 직선 디버깅이 효율적이었다. LWW 머지의 *값별 보존 규칙* 은 sync 도입의 흔한 함정. clothes/fittings/profile 셋 다 같은 패턴으로 일관 적용.

## 종합

오늘 5개 케이스를 통해 다음을 적용/개선했다.

- **분할 커밋 절차의 확장**: 어제 4번의 절차를 6 그룹 / 11 파일 / 한 파일 두 기능 케이스로 확장. 도메인 일치 여부로 한 파일을 한 commit 에 묶는 가이드 추가.
- **plan 갱신의 결정 시점**: Supabase 가 새 비대칭 시스템으로 마이그레이션돼 있다는 현장 상태를 phase 1 시점에 발견 → SUPABASE_JWT_SECRET 흐름을 JWKS 비대칭 흐름으로 전환. plan 따라가기 vs plan 비틀기 결정에서 빠르게 후자.
- **양파형 디버깅의 단계 분리**: Google OAuth 흐름의 4단 (Site URL fallback → 화이트리스트 분리 → maybeCompleteAuthSession → SecureStore chunking) 을 한 풀이로 합치지 않고 한 단계씩 짚어 land. 서버 측 응답을 `curl` 로 검증해 클라/서버 책임을 가르는 단계가 핵심.
- **Windows uvicorn ghost worker 표준 절차**: TaskStop 응답 신뢰 X → tasklist + taskkill /F → `--reload` 없이 재시작. 한번 만들어둔 절차로 같은 상황 재발 시 즉시 통과.
- **LWW 머지의 값별 보존 규칙**: clothes / fittings / profile 셋 다 file:// imageUri 보존. 추가로 tryOn 호출 시점의 robust 변환 패턴. ADR-012 / ADR-014 에 명시.
